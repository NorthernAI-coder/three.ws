# pump.fun v2 trade-interface audit — three.ws (follow-up)

**Date:** 2026-06-11
**Scope:** re-audit of every place three.ws builds a bonding-curve buy/sell, the AMM
swap path, the buyback inner instruction, and the launch (create + initial buy) path —
checked against the vendored v2 docs and IDLs in `docs/pumpfun-program/` and the
**installed** SDK sources (`@pump-fun/pump-sdk@1.36.0`, `@pump-fun/pump-swap-sdk@1.17.0`).
**Prior audit:** [AUDIT-2026-06-08.md](AUDIT-2026-06-08.md). That audit verified account
lists and quote-mint plumbing. This pass re-verified those conclusions on the pinned
1.36.0/1.17.0 SDKs and went one level deeper into **argument semantics** (token amounts,
slippage units, token programs), where it found a cluster of real, reachable defects the
prior pass missed.

## Method

- Account lists / order / flags / discriminators extracted programmatically from
  `idl/pump.json` and compared to our builders and to the SDK builder internals read
  from `node_modules/@pump-fun/pump-sdk/dist/esm/index.js` and
  `node_modules/@pump-fun/pump-swap-sdk/dist/esm/index.js`.
- Argument semantics taken from `docs/instructions/BUY.md` ("`amount` … Must be greater
  than `0`"), `docs/instructions/SELL.md` (`min_sol_output` floor), and
  `docs/instructions/COIN_CREATION.md` (`create_v2` mints are **Token-2022**).
- Three SDK facts drive most verdicts below (verified in the installed sources):
  1. **Slippage unit is PERCENT.** pump-sdk pads via
     `amount * floor(slippage * 10) / 1000` (so `slippage: 1` = 1%, and any value
     < 0.1 floors to **zero**); pump-swap-sdk uses `1 ± slippage / 100`.
  2. **`buy`/`buy_v2` `amount` is the base-token quantity to buy** and must be > 0;
     the SDK passes it through untouched. The quote/sol amount is only the max-cost
     bound. A caller that wants "spend X quote" must derive `amount` via
     `getBuyTokenAmountFromSolAmount`.
  3. **Token programs are caller-supplied.** `fetchBuyState`/`fetchSellState` and the
     legacy builders default to SPL Token; the v2 builders default the base to
     Token-2022. Neither inspects the mint. Since **all** pump.fun launches now go
     through `create_v2` (Token-2022 base mints — including every coin three.ws
     launches), any path that hardcodes/defaults the token program is wrong for one
     class of coin.

## Gaps found and fixed (this pass)

### GAP-1 (critical) — curve buys requested **zero tokens**

`handleBuyPrep` passed `amount: new BN(0)` to `buyV2Instructions` (USDC branch) and
`buyInstructions` (SOL branch). Per `docs/instructions/BUY.md`, `amount` is the base-token
quantity to buy and **must be > 0** — the program rejects the instruction, so every
bonding-curve buy built by `/api/pump/buy-prep` failed at execution. The same defect
existed in the browser skill's USDC buy and in the `pumpfun-create` dev buy
(`createV2AndBuyInstructions` with `amount: 0`). This predates the prior audit (verified
against commit `73e8f91a`) and was missed by it.

**Fix:** every buy path now derives the token amount from the quote input via
`getBuyTokenAmountFromSolAmount({ global, feeConfig, mintSupply, bondingCurve, amount, quoteMint })`
(with the real fetched `feeConfig` where available) and passes the user's quote amount as
the max-cost bound the SDK pads with slippage. Responses now include
`expected_tokens_out`. Zero-token estimates return a typed 400 `amount_too_small`.

### GAP-2 (high) — base token program assumed SPL; create_v2 coins are Token-2022

All pump.fun coins are now minted via `create_v2` with **Token-2022** base mints
(`docs/instructions/COIN_CREATION.md`; our own `buildLaunchInstructions` comment: "Pump.fun
no longer accepts V1 launches"). Yet:

- the dispatcher SOL curve path used legacy `buyInstructions`/`sellInstructions` with the
  default SPL `tokenProgram` and called `fetchBuyState`/`fetchSellState` without a token
  program (SPL ATA lookups);
- the custodial endpoints hardcoded `tokenProgram: splToken.TOKEN_PROGRAM_ID`;
- the custodial portfolio + `pumpfun-status` derived SPL-only ATAs.

Result: wrong `associated_user` ATAs and wrong `token_program` account for every
Token-2022 coin — i.e. **trades of every coin three.ws itself launches failed on-chain**,
and Token-2022 sells fell through to the AMM path (`fetchSellState` threw on the wrong
ATA) and 404'd `pool_not_found`.

**Fix:** all trade handlers read the mint account's owner and resolve the token program
through the new validated helper `resolveTokenProgramForMintOwner`
([api/_lib/pump-trade-args.js](../../api/_lib/pump-trade-args.js)); the quote token
program is likewise read from the quote mint's owner instead of assumed. The dispatcher
curve paths were migrated from the legacy `buy`/`sell` builders to the **unified
`buy_v2`/`sell_v2`** builders for all coins (SOL- and USDC-paired, SPL and Token-2022) —
exactly the migration the upstream announcement urges, with documented identical costs,
and the v2 sell builder also reads the curve's mayhem flag itself (the legacy sell needed
`mayhemMode` passed and we never passed it — mayhem-coin sells picked the wrong fee
recipient pool). Portfolio balances now check both ATA derivations.

### GAP-3 (high) — slippage passed as a fraction; SDKs take percent

Every handler converted `slippage_bps / 10_000` (fraction) and passed it to SDK builders
that expect **percent**. On the curve builders, `floor(slippage * 10)` truncates any
fraction < 0.1 to **zero** — so the live default of 100 bps produced a 0% max-cost pad
(buys revert on any upward tick) and, combined with GAP-4, sells had no floor at all. On
the AMM, bounds came out 100x tighter than requested. Affected: dispatcher buy/sell/quote,
custodial buy/sell/swap, browser skills, `pump-swap-quote.js`, `pump-fun-mcp.js`.

**Fix:** `slippagePercentFromBps()` (clamped, default-aware) used at every SDK call site;
unit equivalence with both SDK formulas is unit-tested. `tests/pump-swap-quote.test.js`
assertions updated from the wrong unit to the verified one.

### GAP-4 (medium) — sells carried `min_sol_output = 0`

`handleSellPrep` passed `solAmount`/`quoteAmount: new BN(0)`; the SDK computes the floor
as `quoteAmount − slippage`, so the program-level minimum output was 0 and the
user-supplied `slippage_bps` was silently ignored (unbounded sandwich exposure).

**Fix:** expected output computed via `getSellSolAmountFromTokenAmount` (real
`feeConfig`) and passed as the quote amount, giving a true `expected − slippage%` floor.
Responses now include `expected_sol_out`/`expected_usdc_out`.

### GAP-5 (low) — stable-quote sells missing the seller's quote ATA

`sell_v2` pays proceeds into `associated_quote_user`, which the program does **not**
initialize (`SELL.md` #16 has no init), and the TS `sellV2Instructions` (unlike the Rust
SDK) prepends no ATA create. A seller without a USDC ATA reverted.

**Fix:** dispatcher and browser sell paths prepend
`createAssociatedTokenAccountIdempotentInstruction` for the quote ATA on non-legacy
quotes.

### GAP-6 (low) — quote endpoint response fields that never existed

The AMM buy quote read `r.uiBase`/`r.minBase` from `buyQuoteInput`, which returns
`{ base, maxQuote, internalQuoteWithoutFees }` — `min_tokens_out` was always null; the
sell path's first fallback `r.quote` likewise never exists. Curve estimates passed
`feeConfig: null` and `mintSupply: null` despite having the fetched curve.

**Fix:** buy quotes return `tokens_out` + the real slippage bound `max_sol_in`/
`max_usdc_in`; sell path reads `uiQuote`/`minQuote`; curve quotes pass the fetched
`feeConfig` and `curve.tokenTotalSupply`.

### GAP-7 (high, dead feature) — AMM browser skills called nonexistent SDK methods

`pumpfun-amm-buy`/`pumpfun-amm-sell` invoked `swapAutocompleteBaseFromQuote` /
`swapAutocompleteQuoteFromBase` on `OnlinePumpAmmSdk` — methods absent from
pump-swap-sdk@1.17.0 (verified by enumerating the prototype) — and passed
`canonicalPumpPoolPda(mint)[0]` (the function returns a `PublicKey`, so `[0]` is
`undefined`). Both skills threw on every call.

**Fix:** rebuilt on the supported surface — `swapSolanaState(poolKey, user)` +
offline `buyQuoteInput`/`sellBaseInput` with percent slippage — the same pattern as the
dispatcher AMM path. Also fixed `pumpfun-status` (`bondingCurveMarketCap` args + dual-ATA
balance) which previously threw for every mint.

### Verified-correct findings worth recording

- **`canonicalPumpPoolPda` version split.** pump-sdk@1.36.0's export takes only `mint`
  (wSOL-keyed) with a separate `canonicalPumpPoolPdaWithQuote`; pump-swap-sdk@1.17.0's
  export takes `(mint, quoteMint = NATIVE_MINT)`. `getAmmPoolState`
  ([pump.js:236](../../api/_lib/pump.js#L236)) imports from **pump-swap-sdk**, so the
  two-arg USDC call is correct. Anyone "simplifying" that import to pump-sdk would
  silently break USDC pools — hence the conformance tests.
- **`buyV2Instructions`/`sellV2Instructions` have no `quoteMint` param** (1.36.0
  signature: `tokenProgram`, `quoteTokenProgram`; quote mint derived from
  `bondingCurve.quoteMint`). The stray `quoteMint:` arguments the old code passed were
  ignored; removed in the rewrite.
- **Legacy `buy`/`sell` builders now append `bondingCurveV2Pda` + `buybackFeeRecipient`
  as remaining accounts** (SDK 1.36.0) — one more reason the unified v2 migration is the
  stable target rather than patching the legacy path.

## Decision: USDC-quoted ("stable paired") coins

The 2026-06-08 decision stands: **adopted** — USDC pairing is live for agent-token
launches (SOL/USDC selector, default USDC) and the quote-asset plumbing (curve + AMM +
launch + confirm) is quote-agnostic. This pass materially hardened that support:

- USDC curve **buys now request a real token amount** (GAP-1) — previously they could
  not execute at all, so USDC trading was wired but non-functional.
- USDC **sells get a real min-out floor** (GAP-4) and a guaranteed **quote ATA** (GAP-5).
- `quote_token_program` is read from the USDC mint's owner rather than assumed (future
  Token-2022 stables keep working).
- USDC mints remain runtime-supplied (`SOLANA_USDC_MINT`/`_DEVNET`, explicit
  `quote_mint`, or `quote_currency: 'usdc'`); no non-`$three` coin is referenced
  anywhere (synthetic keys in tests).

**Remaining productization (unchanged from the prior audit, still open):**

1. Buy/sell widget USDC denomination (`src/game/coin-buy.js` is SOL-only; the API
   already accepts `usdc_amount` end-to-end).
2. `pump_agent_trades.quote_amount`/`quote_mint` columns so USDC trades record amounts
   (USDC buys still store `sol_amount = null`).
3. Portfolio per-quote-asset subtotals.
4. **Custodial USDC trading (new flag from this pass):** `/api/agents/:id/pumpfun/{buy,sell,swap}`
   are SOL-denominated; they now return a typed `409 usdc_paired_coin` for stable-paired
   coins instead of building a doomed transaction. Supporting them custodially needs a
   product decision on USDC accounting for agent wallets (the wallets *do* earn x402
   USDC, so this is a natural follow-up) — flagged, not half-implemented.
