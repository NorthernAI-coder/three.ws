## Method

Ran the repo's existing audit tooling (these are the source of truth, not
guesswork) plus a forbidden-pattern source scan and per-file inspection of the
flagged sites:

## Top-line conclusion

**The platform is in excellent shape.** First-party source is clean: no mocks,
no fake data, no stubs, no `TODO`/`not implemented` markers, no fake loaders.
Type-checking is green and all 872 API handlers and 103 page routes pass their
structural audits. The only substantive code defects found are twelve latent
`ReferenceError`s inside the vendored `character-studio` fork (Finding 1), all
fixed in this pass. The remaining items are lint hygiene and tooling DX.

## Finding 1 — character-studio: 12 latent ReferenceErrors (FIXED)

`character-studio/` is a vendored fork of
[M3-org/CharacterStudio](https://github.com/M3-org/CharacterStudio) (the avatar
builder behind `/create/studio`). It carried twelve `no-undef`/out-of-scope
references — each one throws `ReferenceError` the moment its code path runs.
They sit in user-reachable paths: NFT lock checks, trait lookup, wallet
collection ownership, VRM0 root-node setup, EVM chain switching, GLB/VRM export,
Solana transaction error handling, and KTX2 texture decoding. All twelve are
now fixed.

| File | Was | Now | Effect of the bug |
|---|---|---|---|
| `manifestDataManager.js:67` | `…?.isNFTLocked(testWallet)` | `…?.isNFTLocked()` | `isManifestByIndexNFTLocked()` threw; `isNFTLocked()` takes no arg and the other call sites already call it with none |
| `manifestDataManager.js:70` | `…?.isNFTLocked(testWallet)` | `…?.isNFTLocked()` | same, by-identifier path |
| `manifestDataManager.js:238` | `manifestData[identifier]?.getTraitOptionById(traitID)` | `manifestData.getTraitOptionById(traitID)` | `getTraitOptionById(traitID, optionalIdentifier)` threw; `identifier` was undefined and `manifestData` was already the resolved manifest (matches sibling `getTraitOption` at :216) |
| `manifestDataManager.js:246` | `traitOption = null` | `let traitOption = null` | undeclared assignment → `ReferenceError` in strict-mode ESM; the parallel block at :225 correctly uses `let` |
| `manifestDataManager.js:399` | `manifestData.getModelTraits(traitType)` | `manifestData.getModelTraits(groupTraitID)` | `getModelTraits(groupTraitID, optionalIdentifier)` threw; `traitType` was undefined. The else-branch at :410 confirms `groupTraitID` is the right arg |
| `mint-utils.js:292` | `network.toLowerCase() == ethereum` | `network.toLowerCase() === 'ethereum'` | `connectWallet('ethereum'\|'polygon')` threw before it could switch chains; `ethereum` was a bare identifier, not the string |
| `mint-utils.js:157,178` | `const connection` inside `try`, used in `catch` | hoisted to `let connection;` before `try` | `const` is block-scoped to the `try`, so the `error.signature` recovery path in `catch` threw a second `ReferenceError` while handling the first error |
| `vrmManager.js:3,71` | `addChildAtFirst` used, never imported | added to the `./utils` import | `_setupVRM0()` threw on every VRM0 model load; the helper exists in `utils.js` and is imported correctly in `characterManager.js` |
| `walletCollections.js:58` | `ownsCollection(wallet, network, …)` | `ownsCollection(wallet, chainName, …)` | `checkForOwnership()` threw; `network` was undefined, the parameter is `chainName` |
| `download-utils.js:85` | `getGLBData(model, options)` used `optimized`, never destructured | added `const {optimized = true} = options;` | `getGLBData()` threw; its sibling `getGLBBlobData` (:69) destructures `optimized` correctly |
| `CharacterManifestData.js:877` | `getTraitByIndex(lockFilter = true)` → `collection[index]` | `getTraitByIndex(index, lockFilter = true)` | `TraitModelsGroup.getTraitByIndex()` could never return the right trait; `index` was undefined. The four sibling overloads correctly take `index`; no external callers, so the signature fix is safe |
| `ktx.js:8` | `LIBKTX !== undefined` | `typeof LIBKTX !== 'undefined'` | reading a never-declared global throws `ReferenceError`; the `typeof` guard is the idiomatic optional-global check and falls through to the existing "ktx library undefined" error path when absent |

**Process note:** an initial pass fixed six of these; a follow-up
`no-undef`-only eslint sweep over `character-studio/src` surfaced the other six
(`manifestDataManager` trait/model lookups, the `try`/`catch` scope leak in
`mint-utils`, the missing `optimized` destructure, and the KTX global guard).
The `connection` and `optimized` references were **not** false positives as
first assumed — they are real bugs in *different* methods than the ones first
inspected. The lesson: trust the linter's line numbers over a spot-check.

## Finding 2 — eslint is not CI-practical in this environment (DX)

`npm run lint` (`eslint .` across the whole monorepo) does not complete within a
4-minute window, and even linting a handful of files is startup-bound to
>90s in this dev container. The root config
([eslint.config.js](../../eslint.config.js)) is already correctly scoped (it
ignores `node_modules`, `dist`, vendored trees, and sub-projects with their own
flat config), so this is environmental startup cost, not a config bug.

**Impact:** lint is effectively a manual, scoped-only operation today; there is
no fast `lint` feedback loop and no green/red lint gate in the standard suite.

**Recommendation (not yet applied — needs a perf decision):** add a scoped,
cache-backed lint script (`eslint <paths> --cache --cache-location
node_modules/.cache/eslint`) and/or split lint into per-area scripts mirroring
the existing `audit:*` pattern, so CI can run an incremental lint over changed
paths rather than the full tree. This is a tooling change with a perf tradeoff,
left as a recommendation rather than shipped blind.
