## 1. Mission

The end user enters `/club` for **a 3D social night out where money is real**. They pick an avatar, walk through an entrance, pay an on-chain cover charge (USDC via x402), step onto a pole-stage venue, and tip dancers with real micro-payments that trigger live routines, move a leaderboard, and float reactions for everyone to see. This surface exists to prove that three.ws can host a *living, economically-real 3D space* — entertainment with on-chain stakes — not a static demo.

"Gamechanging" here means a venue that feels **alive and worth returning to**: a place with presence (other people, real or agent), persistent identity and status, social rituals worth paying for, and a real-time economy where every tip is visible and consequential. The dream: someone shows up for two minutes to test it and stays because it's *happening*.

No reason to return tomorrow: no events, no schedule, no leaderboard seasons.

## 4. Innovation directions (seeds — you are expected to go beyond them)

- **Real co-presence:** show other live visitors (and/or platform agents as ambient crowd) moving in the venue via the existing SSE/real-time layer, with avatars, emotes, and proximity — so the room is visibly *occupied*. Solve the empty-room problem first; everything else compounds on it.
- **An escalating tip economy, not a vending machine:** crowd hype meter that everyone's tips raise together to unlock a special routine; tip requests / dedications with on-stage shout-outs; tipping streaks and combos; "top tipper of the night" spotlight. Make each tip part of a shared rising moment, not an isolated transaction.
- **Persistent identity + status:** carry rank, lifetime tips, VIP/regular status, and unlockables across visits (tied to wallet), with leaderboard *seasons* that reset to give newcomers a shot. Status is the reason to return.
- **Scheduled events + a reason for tonight:** a venue schedule (sets, themed nights, agent guest performers from the marketplace), a live "now playing" state, and notifications — so `/club` has a *when*, not just a *where*.
- **Cross-feature wiring (required):** the dancers and crowd should be real platform agents — pull performers from **`/api/agents`** / the marketplace so an agent created in `/brain` or profiled at **`/agents/:id`** can *perform* here, and link tip recipients back to their profile and live $THREE/pump token chip (`/api/pump/by-agent`). Use the extracted Talking Head / lip-sync engine so dancers can *speak* shout-outs. Surface the club in **`/labs`** and let `/agents/:id` deep-link "perform at the club."

> These are starting points, not a checklist. The best idea may not be listed — find it. Think second-order: how does improving this unlock value in adjacent three.ws features? **Wire those connections.** The best platforms feel like everything is linked.

## 5. Hard rules (non-negotiable — from `CLAUDE.md`)

- **Real APIs, real data, real integrations. No mocks, no fake/sample arrays, no placeholders, no `setTimeout` fake-loading or fake progress.** If credentials are missing, find them in `.env` / `.env.example` / `vercel env` — then proceed.
- **Wire 100%.** Every button works, every link goes somewhere, every reachable state exists. Design *every* state: loading (skeletons over spinners), empty (tell the user what to do next), error (actionable recovery), populated, and overflow (0 / 1 / 1000 items, very long strings, mid-operation network failure, expired session).
- **No TODO comments, no stubs, no `throw new Error("not implemented")`, no commented-out code.** If you write it, finish it.
- **No errors without solutions.** Every error has a root cause; every root cause has a fix. Ship failsafes, not lazy propagation.
- **$THREE is the only coin** (CA `FeMbDoX7R1Psc4GEcvJdsbNbZA3bfztcyDCatJVJpump`). Never name, add, hardcode, link, render, or recommend any other coin anywhere — code, copy, tests, fixtures, commits. The cover/tip economy here is **USDC-denominated x402** (coin-agnostic on-chain payment plumbing), which is allowed; do not introduce any non-$THREE token branding or recommendation.
- **Read before you write.** Match the existing patterns, naming, file organization, and the design tokens in `DESIGN-TOKENS.md`. Consistency compounds.
- **Accessibility + responsive (320 / 768 / 1440) + microinteractions** are part of done, not polish. Semantic HTML, ARIA, keyboard nav, focus rings, sufficient contrast.
- **Performance by default:** lazy-load heavy modules, debounce input handlers, paginate large lists, animate with `transform`/`opacity`. Ship no jank.
- **Changelog:** append a holder-readable entry to `data/changelog.json` for any user-visible change, then run `npm run build:pages` to validate.
