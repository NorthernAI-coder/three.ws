## 1. Mission

The home page is three.ws's single most important surface — the front door every visitor, investor, $THREE holder, developer, and crawler hits first. The end user is a first-time visitor who has never heard "the 3D agent layer of the internet" and needs to *understand and believe it in under ten seconds*. They are trying to answer one question: "Is this real, and is it for me?" The page exists to answer that question not with copy, but with a live, hands-on proof of the entire platform — a visitor should *touch* the product (forge a model, pose a rig, see a live token, watch an agent dance) before they ever click away to a product route.

"Gamechanging" here means the home page stops being a brochure with widgets bolted on and becomes **the single best interactive demo of what three.ws is** — a guided, performant, screenshot-worthy journey where the hero is alive, the proof is real (real `/api/forge`, real Pump.fun data, real `/api/explore`), and every scroll reveals a capability the visitor can immediately use. This is the only deeply interactive content page on the platform; treat it as a flagship product surface, not a landing page.

## 2. The bar

Build the version that makes someone screenshot it and share it. Benchmark against the best in the world — the Linear marketing page (motion + clarity), the Stripe homepage (live, credible product proof), the Vercel homepage (developer trust + speed), the Figma/Rive homepages (interactive canvas as hero), and Apple product pages (scroll-driven storytelling). **If your result merely matches what already exists, you have failed this brief.** Invent something true and new: a home page where the visitor builds something real in the hero before they scroll, and leaves having *used* the platform, not read about it.

## 3. Current state — ground truth (verify in source before you touch anything)

- **Route(s):** `/` → `/home.html` (per `vercel.json`). No auth, fully public.
- **Source:** `pages/home.html` (markup + inline modules); widget modules `src/home-forge.js`, `src/home-pose.js`, `src/home-act2-viewer.js`, `src/home-live-token.js`; lazy mounts listed above; web-component runtime `<script src="https://three.ws/agent-3d/latest/agent-3d.js">` + `/embed.js`; chrome via `/nav.js`, `/footer.js`.
- **Current flow:** ~3 required steps (land → optional browse → click a CTA into a product route), **plus** ~40 distinct CTAs and ~13 self-contained interactive widgets, each an optional sub-flow. The landing flow is browse-and-branch, not linear.
- **What works today:**
  - Hero: live `<agent-3d>` on stage + 7 animation chips (🎲 Random, 👋 Wave, 💃 Dance, 🤸 Capoeira, 🦘 Jump, 🧟 Thriller, 🙏 Pray) with a trigger counter; eyebrow link → `/forge`.
  - Below-the-fold widgets lazy-load on `IntersectionObserver` to keep initial payload light.
  - **Mini Forge** (`#home-forge`, `src/home-forge.js`): real text-to-3D — typewriter prompt suggestions (Tab to accept), example chips, `POST /api/forge` (tier `standard`, TRELLIS lane, free, auto-degrades to NVIDIA), poll `GET /api/forge?job=…`, live `<model-viewer>` result, localStorage history (`forge:home:history`), result toolbar (auto-rotate, variation, Scene Studio, copy share link, GLB download, clear), embed sheet, honest elapsed timer, cancel/retry wired.
  - **Playground** (`#embed`): editable embed-code textarea (HTML/React/Vue tabs), mode/background chips, feature toggles, live preview, 7 test chips, copy button.
  - **Pose Studio** (`#home-pose`), **AR** (`#home-ar`, `<model-viewer>` + lazy QR), **Live token card** (`#hlt-card`, real Pump.fun data + homepage launcher).
  - Live API-fed sections: **Oracle feed** `GET /api/oracle/feed?network=mainnet&limit=6&min_score=56`; **Community Forge gallery** `GET /api/forge-gallery?scope=community&limit=18`; **Showcase 3D grid** `GET /api/explore`.
- **Real APIs / dependencies already wired:** `POST/GET /api/forge`, `GET /api/forge-gallery`, `GET /api/explore`, `GET /api/oracle/feed`; Pump.fun feed (live token card); `agent-3d` runtime + Google `model-viewer` CDN (lazy); Solana RPC indirectly via the launcher.
- **Where it's mediocre, thin, or unfinished:**
  - **One giant 6921-line HTML file.** ~23 sections stacked vertically with ~40 CTAs and ~13 widgets — generous, but the *narrative* is a list, not a story. There is no spine guiding a first-timer from "what is this" → "watch it" → "build it" → "ship it." Visitors must self-navigate a wall of demos.
  - **The hero proves nothing the visitor did.** The agent dances on its own; the visitor watches. The first real *creation* (Mini Forge) is below the fold. The strongest proof (a real forged model in seconds) is buried where many never reach it.
  - **No memory / personalization.** A returning visitor who already forged a model on the home page sees the same cold start; the localStorage history exists but doesn't shape the experience.
  - **Widgets are isolated islands.** The Mini-Forge result, Pose rig, AR viewer, and Showcase grid don't share a model — forge something and you can't immediately pose it, AR-view it, or embed it without leaving. The cross-widget wiring the platform brags about is missing on its own home page.
  - **No scroll-progress / section navigation / share affordances.** With ~23 sections there's no minimap, no "jump to capability," no way to deep-link or screenshot a specific demo.
  - **Performance risk:** many live `<agent-3d>`/`<model-viewer>` instances; verify lazy-mount discipline and that off-screen canvases pause.

## 4. Innovation directions (seeds — you are expected to go beyond them)

- **"Forge in the hero" — the visitor creates in the first 10 seconds.** Move (or mirror) a single-line prompt bar directly into the hero, pre-filled with a rotating real suggestion. Hitting enter forges a real model via `/api/forge` *on the hero stage itself*, replacing the canned dancer with **the visitor's own creation**. Nothing converts like watching your own words become a 3D agent you can spin, before you've scrolled once.
- **One model, every capability — a "creation passport" that follows the scroll.** When the visitor forges (hero or Mini-Forge), persist that model id and feed it into every downstream widget: the Pose rig poses *their* model, the AR card AR-views *their* model, the Playground embed snippet embeds *their* model, the Scene Studio CTA opens *their* model. The whole page becomes a personalized tour of "here's everything you can now do with the thing you just made." This is the ≥1 cross-feature wiring idea — and it makes the platform's "everything is linked" promise true on the front door.
- **Scroll-driven narrative spine + a capability minimap.** Replace the flat section stack with an intentional act structure (Understand → Watch → Build → Monetize → Ship), a fixed scroll-progress rail, and a keyboard-navigable section minimap (press a number / `j`-`k` to jump). Deep-linkable section hashes so any demo is shareable.
- **Live platform pulse in the hero.** A small, real, animated stat strip fed by existing endpoints (`/api/explore` count, `/api/forge-gallery` recent creations, `/api/oracle/feed` top conviction, live $THREE from the token card) — "X agents built today," "newest community forge," updating in real time. Credible, true social proof, no fake numbers.
- **Returning-visitor warm start.** If `forge:home:history` (or the creation passport) has entries, the hero greets the visitor with their last creation already on stage and a "pick up where you left off →" path — first impression for newcomers, continuation for returners.
- **"Watch the platform build itself" cinematic moment.** A reduced-motion-respecting, scroll-triggered sequence where a prompt types, the model forges, gets posed, drops into AR, then renders as an embed snippet — one continuous real artifact flowing through the product, proving the whole pipeline in one screen-height.

> These are starting points, not a checklist. The best idea may not be listed — find it. Think second-order: the home page is where Forge, Pose, AR, Scene Studio, Embed/Playground, Oracle, Launches, and Discover all already appear — wire them so a single creation flows between them. **Wire those connections.** The best platforms feel like everything is linked.

## 5. Hard rules (non-negotiable — from `CLAUDE.md`)

- **Real APIs, real data, real integrations. No mocks, no fake/sample arrays, no placeholders, no `setTimeout` fake-loading or fake progress.** If credentials are missing, find them in `.env` / `.env.example` / `vercel env` — then proceed.
- **Wire 100%.** Every button works, every link goes somewhere, every reachable state exists. Design *every* state: loading (skeletons over spinners), empty (tell the user what to do next), error (actionable recovery), populated, and overflow (0 / 1 / 1000 items, very long strings, mid-operation network failure, expired session).
- **No TODO comments, no stubs, no `throw new Error("not implemented")`, no commented-out code.** If you write it, finish it.
- **No errors without solutions.** Every error has a root cause; every root cause has a fix. Ship failsafes, not lazy propagation.
- **$THREE is the only coin** (CA `FeMbDoX7R1Psc4GEcvJdsbNbZA3bfztcyDCatJVJpump`). Never name, add, hardcode, link, render, or recommend any other coin anywhere — code, copy, tests, fixtures, commits. The only exceptions are coin-agnostic runtime plumbing (user supplies a mint) and platform launch records rendered at runtime.
- **Read before you write.** Match the existing patterns, naming, file organization, and the design tokens in `DESIGN-TOKENS.md`. Consistency compounds.
- **Accessibility + responsive (320 / 768 / 1440) + microinteractions** are part of done, not polish. Semantic HTML, ARIA, keyboard nav, focus rings, sufficient contrast.
- **Performance by default:** lazy-load heavy modules, debounce input handlers, paginate large lists, animate with `transform`/`opacity`. Ship no jank.
- **Changelog:** append a holder-readable entry to `data/changelog.json` for any user-visible change, then run `npm run build:pages` to validate.
- **Concurrent agents share this worktree.** Stage explicit paths only — **never** `git add -A` / `git add .`. Re-check `git status` + `git diff --staged` immediately before any commit. Never commit `api/*.js` esbuild bundles (check `head -1` for `__defProp` / `createRequire`).
