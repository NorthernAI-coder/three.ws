# F3 — Harden CI/CD & Deploy Safety

You are a senior engineer + product thinker building **three.ws**. Read `CLAUDE.md`, `STRUCTURE.md`, and `prompts/production-campaign/00b-the-bar.md` first. **Prerequisites:** F1 (its `db:migrate --verify` and Redis-singleton guard want a CI home) and F2 (worker health / dead-letter checks want a gate) — coordinate, but F3 can land independently.

## Why this matters for $1B
CI/CD is the platform's immune system. Real money rides on this code, so every merge to `main` auto-deploys to Vercel — which means a broken build, a leaked secret, a regressed Lighthouse score, or the documented `npx vercel build` footgun (it overwrites `api/*.js` source with esbuild bundles) ships straight to production. This repo has already taken production down: 465 consecutive 500s for 90 minutes on 2026-06-11 from a committed symlink + an unresolvable peer-dep import — exactly what `scripts/audit-deploy-artifacts.mjs` now catches. A $1B platform never has a class of failure it can ship twice: every known footgun becomes a CI gate, every deploy is preview-verified, and there is always a one-command rollback. This task turns the existing checks-that-exist-but-aren't-enforced into real merge gates and a safe deploy path.

## Your mission
### 1. Promote the audits into real merge gates
Expand `.github/workflows/ci.yml` so the existing-but-unenforced scripts become required checks. Add jobs (or steps in `guards`) for `test:gate`, `audit:pages`, `audit:handlers`, `audit:mcp`, `audit:deploy`, and `check:dist`. Run them in parallel where independent; cache `npm ci` (already `cache: npm`). Each must fail the build on a real regression — no `continue-on-error` on a money/correctness check. Keep `typecheck` advisory until its backlog is burned down (note the path to flip it). Where F1 added a migration `--verify` and a Redis-singleton/critical-bucket guard, wire those in too.

### 2. Make the bundle-overwrite trap un-shippable
`scripts/check-api-not-bundled.mjs` guards committed `api/*.js`. Strengthen the guard end-to-end: confirm it scans **every** `api/**/*.js` (including nested dirs like `api/_lib`, `api/coin`, `api/club`), catches `__defProp`/`createRequire`/`esbuild` banner signatures, and runs both in CI and (ideally) as a pre-commit-style local check documented in `docs/deployment.md`. Confirm `scripts/build-vercel.mjs` / `vercel.json` never write esbuild output back over source in the committed tree. Document the recovery (`git restore -- api/ public/`) prominently in the deploy runbook so an agent who trips it self-recovers in one command.

### 3. Lighthouse / Core Web Vitals gate
Wire `.lighthouserc.json` into CI as a real (initially advisory-then-enforced) job against the primary surfaces, holding the `00b-the-bar.md` budgets (LCP < 2.5s, CLS < 0.1, INP < 200ms). Run it against a preview deploy URL (mission 4) or a local `npm run dev`/`vite preview` build. Start advisory if current scores aren't green, but record the baseline and the gap so the next surfaces agent (Track C) can close it; flip to enforced for any surface already in budget.

### 4. Preview deploys + `audit:deploy` artifacts on every PR
Ensure every PR gets a Vercel **preview deploy** (Vercel's GitHub integration or a workflow that runs `vercel build`/`deploy --prebuilt`) so changes are verified on a real URL before merge, never first-seen in production. Run `audit:deploy` (`scripts/audit-deploy-artifacts.mjs`) as a hard gate — it catches the exact 2026-06-11 class (committed symlinks, unresolvable peer-deps, phantom imports) in seconds. Surface the preview URL + the artifact audit result on the PR. Confirm the smoke checks (`smoke:onchain`, `smoke:mcp`) run against the preview or a safe target, not production money paths.

### 5. A documented, one-command rollback path
There must be a safe, fast rollback for both halves of the system: Vercel (instant promote of the previous good deployment — document the `vercel rollback`/promote-previous flow and how to find the last-green SHA) and the Cloud Run workers (`scripts/deploy-sniper.mjs --skip-build` redeploys the current image; document redeploying a prior revision). Write this into `docs/deployment.md` / `docs/ops/` as a runbook: how to detect a bad deploy (the heartbeat/health from F2, the 500-rate), how to roll back each surface, and how to confirm recovery. The DB rollback convention from F1 (forward-only + paired down-migrations) is referenced here as the data half of rollback.

### 6. Verify it for real
Run the full CI suite locally (`npm run lint`, `npx vitest run`, `npm run test:gate`, `npm run audit:pages`, `npm run audit:handlers`, `npm run audit:mcp`, `npm run audit:deploy`, `npm run check:dist`, `npm run build:pages`) and confirm each passes or surface the real failure. Deliberately introduce a fake bundled `api/*.js` and a committed symlink in a scratch branch (then revert) to prove the guards fire. Confirm the workflow YAML is valid and the new jobs are wired as required checks. Never run `npx vercel build` against the working tree without checking `head -1` of changed `api/` files afterward.
