# Phase 4 · 03 — Decentralized Provider Adapter + Router

You are a senior backend engineer on **three.ws**. Read `CLAUDE.md` and `STRUCTURE.md`
first — CLAUDE.md overrides defaults. Read `api/_providers/gcp.js` closely — you are
writing its decentralized sibling, with the **identical contract**. Read the protocol +
roster APIs from prompt 01.

## What you're building

The piece that lets three.ws's own code send a GPU job to the mesh **without caring that
the GPU is decentralized**, with a router that picks the right node and falls back to the
centralized path the instant the mesh can't deliver. This is what makes the network
usable in production rather than a science project.

### 1. The provider adapter — `api/_providers/decentralized.js`

Implement the exact contract `gcp.js` exposes so it's a drop-in alongside it:

```
submit(request)  → { extJobId, eta }
status(extJobId) → { status, resultGlbUrl?, resultImageUrl?, resultUrl?, error?, node_id?, signed? }
```

- `submit` selects a node (via the router below), writes an `oin_jobs` row, dispatches
  the manifest (push to the node's endpoint, or enqueue for pull-claim), returns the
  base64url `extJobId` envelope (carry `job_id` + `node_id` + `model_key`, same shape as
  `gcp.js`).
- `status` reads job state, **verifies the node signature** on completion via
  `api/_lib/oin/protocol.js` before reporting `done` — an unsigned or bad-sig result is
  treated as a failure and reassigned, never returned to the caller.

### 2. The router — `api/_lib/oin/router.js`

Given `(model_key, params, deadline, max_price)`, choose the best node:

- **Eligibility:** healthy (recent heartbeat), advertises the model with enough VRAM,
  region/latency acceptable, has spare concurrency, price ≤ max.
- **Ranking:** weighted score over recent latency, success rate, and reputation
  (consume the reputation signal from prompt 07 when present; degrade to latency +
  success-rate before it lands). Add light load-spreading so one fast node isn't
  hammered — don't always pick rank #1.
- **Reassignment:** if the chosen node misses the deadline, errors, or returns a bad
  signature, transparently reassign to the next node (bounded retries). Record
  `fallback_used` / reassignment on the job row.

### 3. The fallback chain (non-negotiable — "no errors without solutions")

The mesh must never make a user job worse than it is today. When **no** eligible node
exists, or the bounded reassignment budget is exhausted, the router falls back to the
**existing centralized provider** (`gcp.js` for GPU/3D jobs, the centralized LLM path
for inference) for that job. The user gets their result; the job row records that the
mesh couldn't serve it (this is the data prompt 09 and the network metrics need to track
the "≥50% served by independent nodes" north star).

### 4. Routing policy + flag

- A per-model routing policy (`api/_lib/oin/policy.js` or a config table): `off` |
  `shadow` (run on mesh, but serve the centralized result — for latency/quality A/B) |
  `prefer` (mesh first, centralized fallback) | `only`. Default new models to `shadow`
  so the mesh proves itself before taking user-facing traffic.
- An env/flag gate so the whole network can be toggled per environment without a deploy.

## Hard requirements

- **Same contract as `gcp.js`** — callers (`api/forge.js`, the avatar pipeline) must not
  need special-casing beyond choosing the provider. Prompt 05 wires those callers.
- **No node is trusted** — every completion is signature-verified before it's served.
- **No mock roster** — the router reads real `oin_nodes` / heartbeats. With zero nodes
  registered, it falls back cleanly (and that path is tested).
- Real selection logic, not round-robin-on-a-hardcoded-list.

## Definition of done

- `api/_providers/decentralized.js` passes the same shape-contract tests as `gcp.js`
  (add a shared contract test in `tests/` exercising both providers).
- Router unit tests: eligibility filtering, ranking, reassignment on
  miss/error/bad-sig, and **fallback to centralized when no node qualifies**.
- A job dispatched with `policy: prefer` runs on a real registered node end-to-end and
  returns a verified result; with no nodes, the same call still succeeds via fallback.
- `shadow` mode records mesh-vs-centralized latency/quality without affecting the user.
- `data/changelog.json` entry (infra), `npm run build:pages`, `npm test` green.
- Explicit-path staging; reviewed `git diff`; no bundled `api/*.js`.

If a node dies mid-job and the user never notices, you built it right.
