## What you're building

### 1. Node reputation scoring — `api/_lib/oin/reputation.js`

A composite, **provable** score per node, computed from real job history (not vibes):

- **Latency** vs the model's SLA target. **Success rate** (completed vs claimed-then-
  failed/timed-out). **Result validity** (see verification below). **Uptime** (heartbeat
  continuity). **Volume** (jobs served, as a tie-breaker, not a crutch).
- Windowed and decaying so a node can recover from a bad day and can't coast on old glory
  — mirror the TraderScore windowing already in the codebase.
- **Attested onchain** on a schedule (cron in `vercel.json`, idempotent like
  `trader-score-attest.js`) so the score the router trusts is the score anyone can verify.
- The router (03) consumes this as its ranking weight; the operator console (06) shows it.

### 2. Result verification — the heart of trustlessness

A node's signature proves *who* produced a result, not that it's *correct*. Add
verification so a lazy or malicious node can't return garbage and get paid:

- **Redundant re-run (challenge):** a sampled fraction of jobs (and any flagged job) is
  re-dispatched to a second independent node; compare result hashes / a perceptual
  similarity check appropriate to the modality (exact-match for deterministic outputs,
  a tolerance metric for stochastic mesh/image gen). Mismatch → the result is rejected,
  the user is served the verified one, and the offending node is penalized.
- **Determinism pinning** where the model supports it (seed/params in the manifest) so
  re-runs are comparable.
- **Cheap sanity gates** before expensive re-runs: result is well-formed (valid GLB/PNG/
  token stream), within size/shape bounds, produced within the deadline.

Make the verification rate a tunable policy (higher for new/low-rep nodes, lower for
proven ones) so security cost scales with risk.

### 3. Staking & slashing (cryptoeconomic security)

Tie real money to honest behavior, using the onchain registry from prompt 01:

- A node stakes (USDC / $THREE) at registration; stake size can gate which value of jobs
  it's eligible for (the router won't send a $5 H100 job to an unstaked node).
- **Slashing:** a verified bad result (failed challenge), a signature over a result it
  didn't produce, or sustained SLA violation slashes a portion of stake — split between
  the wronged payer (refund/redo) and the protocol/insurance pool. Implement the slashing
  path against the real contract; document the governance/dispute window.
- **Dispute resolution:** a payer can challenge a result within a window; the challenge
  triggers verification; the loser pays the cost. No he-said-she-said — every dispute
  resolves to a hash comparison.

## Hard requirements

- **No node is trusted by default.** Reputation, verification, and stake are the trust
  model — there is no admin override that just believes a node.
- **Real onchain attestation + slashing**, on the existing rails. Not a database flag
  pretending to be a chain. If a contract method needs deploying, ship the source/ABI and
  the real call, document the deploy.
- Every penalty traces to a verifiable cause (a failed challenge with two hashes, an SLA
  log) — surface that cause to the operator (06) so it's contestable, not arbitrary.

## Definition of done

- Reputation computes from real job history, decays/windows correctly, attests onchain
  on schedule, and drives the router's ranking.
- A sampled job is re-run on a second node; a deliberately-wrong result is caught,
  rejected, the user served the correct result, and the bad node penalized (test with a
  fault-injected node).
- Staking gates job eligibility; a verified bad result slashes stake to the right
  parties; a dispute resolves by hash comparison — all on the real chain path.
- Verification rate scales with node risk and is policy-tunable.
- Tests in `tests/` cover scoring, the challenge/verify path, mismatch handling, and
  slashing accounting. Existing tests pass.
- `data/changelog.json` entry ("Trustless results — node verification, reputation &
  slashing live"), `npm run build:pages`, `npm test` green.
- Explicit-path staging; reviewed `git diff`; no bundled `api/*.js`.

The promise is "decentralized, and *more* provable than a black-box API." Deliver the
provable part.
