# Phase 4 · 06 — Node Operator Console + Live Network Map

You are a senior product engineer on **three.ws**. Read `CLAUDE.md` and `STRUCTURE.md`
first — CLAUDE.md overrides defaults. This is the **face** of the Open Inference
Network: where a person decides to run a node, registers it, and watches it earn — and
where anyone can see the mesh is real. Match the polish of the rest of the platform
(study `pages/leaderboard.html`, `pages/vaults.html`, the dashboard, and `DESIGN-TOKENS.md`).

Consume the registry/roster/heartbeat APIs (01), the earnings ledger (04), and the
traffic accounting (05). Frontend is vanilla JS modules + Vite, mobile-first.

## What you're building

### 1. "Become a node" onboarding — the operator funnel

A page (`pages/run-a-node.html` + its module) that turns a GPU owner into an operator:

- **The pitch:** what the network is, what hardware earns what, the `docker run`
  one-liner from prompt 02. Real numbers from the pricing catalog (04), not invented ones.
- **Register flow:** connect wallet (the payout address), generate/paste the node key,
  submit the capability advert, complete the onchain registration. Wire it to
  `POST /api/oin/register` — real chain write, every state designed (connecting,
  signing, pending confirmation, registered, error-with-fix).
- **Health check helper:** surface the node's `GET /health` + heartbeat status so an
  operator immediately sees whether their box is actually serving.

### 2. The operator dashboard — "my node"

For a registered operator (authenticated by node key / wallet):

- **Live status:** online/offline, healthy models, current load, last heartbeat, version.
- **Earnings:** gross / fee / net over time, by model, by day (from `/api/oin/earnings`),
  each settlement linking to its onchain tx and receipt. A "withdraw"/payout-address view.
- **Jobs served:** recent jobs, latency, success/fail, reputation trend (from 07 when
  present). An operator should be able to debug *why* a job failed from here.
- **Reputation & SLA:** the node's standing and what improves it (latency, uptime,
  result validity) — make the incentive legible.

### 3. The live network map — the shareable artifact

The piece worth screenshotting (see the visual in
`content/whats-next/later/12-decentralized-gpu-inference.md`): a public, real-time view
of the mesh.

- A globe / node graph of registered nodes by region (real data from `GET /api/oin/nodes`
  — never a decorative fake), pulsing as jobs route. Total nodes, models served, jobs in
  the last 24h, **% of traffic served by independent nodes** (the README north star,
  live). Use Three.js/WebGL — this is a 3D-native platform; make it beautiful and
  performant (lazy-load, `transform`/`will-change`, no jank).
- Degrade gracefully: with few/zero nodes online it still reads as a real, honest network
  ("3 nodes online, be the 4th") — not an empty void and not a faked swarm.

### 4. Cross-link it

Surface "Run a node" from the dashboard / footer / docs nav. Link node profiles from the
network map. Wire the network-map stats into wherever the platform shows its
infrastructure story. The best platforms feel like everything is connected — connect this.

## Definition of done

- A real GPU owner can land on `run-a-node.html`, register a node onchain, and see it
  appear on the live map and in their dashboard — no human help needed.
- Operator dashboard shows real status + real earnings with onchain links; every state
  (loading skeletons, empty, error, populated, overflow) designed; works at 320/768/1440.
- Network map renders real roster data, animates job routing, shows the live independent-
  traffic % , and is performant and shareable.
- Hover/active/focus states on every interactive element; semantic HTML; ARIA; keyboard
  nav; visible focus rings; sufficient contrast.
- No console errors/warnings; network tab shows real API calls; exercised in a browser
  (`npm run dev`).
- `data/changelog.json` entry ("Run a node and watch it earn — operator console + live
  network map"), `npm run build:pages`. If it's a new page, the `data/pages.json` `added`
  date feeds the changelog automatically — register it there. `npm test` green.
- Explicit-path staging; reviewed `git diff`.

Make someone want to plug in their GPU. That's the job.
