# Phase 4 · 04 — Onchain Settlement & Node Payouts (x402 receipts)

You are a senior payments engineer on **three.ws**. Read `CLAUDE.md` and `STRUCTURE.md`
first — CLAUDE.md overrides defaults. Read `api/_lib/x402-paid-endpoint.js`,
`api/_lib/x402-spec.js`, a few `api/x402/*` endpoints, and the receipt schema from
prompt 01. This is the money loop: **agents pay nodes onchain for compute, with a
cryptographic receipt for every job.**

## What you're building

The settlement layer that turns a signed job result into a provable onchain payout to
the node operator — pay-per-job (and pay-per-token for the LLM leg), denominated in
**USDC and/or $THREE only**. This is the README deliverable "onchain settlement for
inference jobs — pay-per-token with cryptographic receipts."

### 1. Per-job pricing — `api/_lib/oin/pricing.js`

A real pricing table keyed by `model_key` and hardware/quality tier, reusing
`api/_lib/pricing/catalog.js` conventions. GPU/3D jobs price per job (with a tier
multiplier: draft/standard/high, L4 vs H100). The LLM leg prices per token (input +
output), mirroring the `upto` metered scheme already in `x402-spec.js`. A node's
advertised price must be ≤ the catalog ceiling for the job to route to it (the router in
03 already enforces `max_price`); the operator keeps the bulk, the protocol takes a small
platform fee — make the split explicit and configurable, not magic numbers.

### 2. The settlement flow

When `decentralized.js` reports a verified `done`:

1. Assemble the **receipt** from the protocol schema (job_id, node_id, payer, model_key,
   price, result_hash, node_sig, ts) and sign it EIP-712 / JWS via `x402-spec.js`.
2. Settle the payer → node payout onchain using the **existing** settlement backends —
   Base via Coinbase CDP, BSC direct, Solana direct (`api/_lib/x402-*` already wires
   these). The node's payout address comes from `oin_nodes`. Settle to the node's chain.
3. Persist the receipt + settlement tx ref on the `oin_jobs` row and in an
   `oin_receipts` table (migration + `schema.sql`). Idempotent on `job_id` — a retried
   settlement never double-pays (reuse the idempotency-key pattern from `api/x402/*`).
4. The platform fee accrues to the protocol treasury; the rest goes to the operator.

### 3. The paid entry point

For external agents calling the network directly (not just three.ws's own jobs), expose
`api/oin/infer.js` (or `api/x402/oin-compute.js`) built with `paidEndpoint(spec)`: the
caller pays the 402 challenge up front, the router dispatches, the node serves, the
result returns, and settlement to the node happens against the collected payment. This is
how an arbitrary agent buys decentralized compute with x402 — wire the full 8-step flow,
register it in the bazaar discovery catalog like the other `api/x402/*` endpoints, and
support both `exact` (fixed per-job) and `upto` (metered LLM) schemes.

### 4. Node earnings ledger

- `GET /api/oin/earnings?node=<id>` — a node's settled jobs, gross, fee, net, by day and
  by model. Authenticated with the node key. Feeds the operator console (06).
- Reconciliation: a scheduled job (cron in `vercel.json`, idempotent like
  `api/cron/rewards-distribute.js`) that catches any verified-but-unsettled job and
  settles it — settlement must be eventually-consistent and self-healing, never a job
  that silently never paid.

## Hard requirements

- **Real onchain settlement.** No mock facilitator, no fake tx hashes. Use the wired CDP
  / BSC / Solana paths. If a chain's creds are absent locally, the code path is real and
  you say which env it needs — you don't stub it.
- **Idempotent, no double-pay.** Prove it with a test that retries settlement.
- **USDC / $THREE only.** Pricing, receipts, ledgers reference no other token, ever.
- Every payout traces to a signed result and a signed receipt — no result, no money.

## Definition of done

- A verified job settles onchain to a real node payout address and writes a receipt +
  tx ref; the receipt verifies against `x402-spec.js`.
- `api/oin/infer.js` is a working `paidEndpoint`: an external x402 caller pays, gets a
  mesh-served result, and the node gets paid — end to end, in the bazaar catalog.
- Retried settlement does not double-pay (test). Reconciliation cron settles a stranded
  verified job (test).
- Earnings endpoint returns real settled data with every state designed.
- Migration + `schema.sql` updated; `vercel.json` route + cron registered.
- `data/changelog.json` entry ("Get paid onchain for running a node — per-job settlement
  live"), `npm run build:pages`, `npm test` green.
- Explicit-path staging; reviewed `git diff`; no bundled `api/*.js`.

A node operator who serves a job must be able to point at a block explorer and see their
money. That's the bar.
