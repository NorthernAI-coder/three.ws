# Phase 4 · 02 — Node Operator Client (the Docker GPU node)

You are a senior systems engineer on **three.ws**. Read `CLAUDE.md` and `STRUCTURE.md`
first — CLAUDE.md overrides defaults. Read `docs/open-inference-protocol.md` and
`api/_lib/oin/protocol.js` from prompt 01 — you implement against that spec exactly.

## What you're building

**The software an independent operator runs on their own GPU box to earn money serving
three.ws jobs.** This is the heart of decentralization: anyone with an A100/H100/L4 can
`docker run` this and become part of the mesh. It must work on a real machine, today.

Create a new top-level worker package — `workers/oin-node/` — that follows the existing
`workers/` conventions (Dockerfile + cloudbuild-style build, FastAPI or a lean Python
service, `GET /health`). It reuses the **existing model code already in this repo** —
do not rewrite inference. The node is an orchestrator wrapping `workers/model-*`,
`workers/texture`, `workers/rembg`, etc.

### The node lifecycle

1. **Boot & capability detection.** Detect GPU (name, VRAM via `nvidia-smi`/`torch`),
   decide which model keys it can serve given VRAM (a 24 GB L4 ≠ an 80 GB H100 — gate
   the heavy mesh/video models accordingly). Generate or load the node keypair (EVM
   secp256k1 and/or Solana ed25519, per the protocol).
2. **Onchain registration.** Register via `api/_lib/oin/registry.js` (or call
   `POST /api/oin/register` with the onchain ref). Real chain write — funded by the
   operator. Document the one-time `oin-node register` step in the README.
3. **Model loading.** Lazy-load each advertised model's weights on first job (reuse the
   loaders from the corresponding `workers/model-*` service — import or vendor them,
   don't fork the inference logic). Report `healthy_models` only for weights that
   actually loaded. Stage weights with the existing `workers/deploy/stage-weights.sh`
   pattern.
4. **Job loop.** Two modes, both from the protocol — implement at least **pull**:
   - **Pull:** poll `GET /api/oin/jobs/claim?node=<id>` (signed request) for a job
     matching its healthy models, atomically claim it, run it, return the result.
   - **Push:** expose an authenticated endpoint the router can `POST` a manifest to.
   Respect `deadline_ms` and `max_price_atomics`; reject jobs it can't meet in time
   rather than starting and timing out.
5. **Execute & sign.** Run the job through the matching model code, hash the result
   (upload the artifact to the operator's configured storage — R2/GCS/S3 — or stream it
   back per the protocol), and **sign the response** with the node key over the canonical
   hash from `protocol.js`. Emit the receipt.
6. **Heartbeat.** Periodic `POST /api/oin/heartbeat` with load, healthy models, recent
   latency. Back off and re-register on network errors — never crash-loop silently.

### Operator ergonomics (this is a product, treat it like one)

- A clean `README.md`: hardware requirements per model tier, `docker run` one-liner,
  env vars (node key, payout address, chain RPC, storage creds), the register command,
  and a realistic earnings/cost note.
- A `oin-node doctor` / startup self-check that prints exactly what's wrong if GPU,
  weights, keys, RPC, or storage are misconfigured — actionable, not a stack trace.
- Structured logs: jobs claimed, served, signed, paid; errors with cause + fix.
- Graceful shutdown: finish in-flight jobs, deregister/mark-offline, don't strand a payer.

## Hard requirements

- **No simulated inference.** The node runs the real models. If you can only test a
  CPU-capable path locally (e.g. `rembg`/`segment`/`remesh`), do that and say so — never
  fake a GPU result.
- **No shared central secret as the trust root** — the node authenticates with its own
  key. That's the protocol.
- Signature and hashing match `api/_lib/oin/protocol.js` byte-for-byte (cross-test
  against the JS verifier from prompt 01).

## Definition of done

- `workers/oin-node/` builds into a Docker image; `GET /health` reports GPU + healthy
  models; boots on a real GPU host.
- A real node registers onchain, appears in `GET /api/oin/nodes`, claims a job, runs it,
  returns a **signed** result the prompt-01 verifier accepts.
- Tampering with the result fails verification; an over-deadline job is declined cleanly.
- README lets an operator go from bare box → earning node without asking a human.
- A test (in `workers/oin-node/` and/or `tests/`) covers capability detection, the claim
  loop state machine, and the sign/verify round-trip.
- `data/changelog.json` entry ("Run a GPU node on the Open Inference Network — node
  client released"), `npm run build:pages`. `npm test` green.
- Explicit-path staging; reviewed `git diff`.

This is the on-ramp for every node operator. If it isn't dead-simple to run and
impossible to spoof, it isn't done.
