# Phase 4 · 08 — Federation + the Open SDK

You are a senior platform engineer on **three.ws**. Read `CLAUDE.md` and `STRUCTURE.md`
first — CLAUDE.md overrides defaults. The mesh works and is trustworthy; now make it
**open** — federate with existing decentralized compute networks so the mesh has capacity
from day one, and publish an SDK so any external agent can discover, pay for, and use
decentralized compute through one interface. README deliverable: "federation with existing
decentralized compute networks where appropriate."

Read `api/inference/livepeer.js` (the existing federation seed), the provider/router (03),
settlement (04), the published-SDK conventions in `STRUCTURE.md` and the existing
`sdk/` / `*-sdk/` packages, and the bazaar discovery pattern in `api/x402/*`.

## What you're building

### 1. Federation adapters — one interface, many networks

Generalize the Livepeer experiment into a pluggable federation layer so the router (03)
can route a job to a three.ws node **or** to an external decentralized network,
whichever best serves it:

- `api/_lib/oin/federation/` with one adapter per network, each implementing the same
  `submit`/`status` contract as `api/_providers/decentralized.js`:
  - **Livepeer** — promote `api/inference/livepeer.js` into a proper adapter (LLM and,
    where available, image/video gateways). Keep the side-by-side comparison demo working.
  - At least one more real network adapter where it genuinely adds capacity for a model
    three.ws serves (e.g. a decentralized GPU/render network with a real public API).
    Only integrate networks with a real, callable API — no aspirational stubs.
- The router treats federated capacity as additional "nodes" with their own latency,
  price, and reliability signals, subject to the same policy (`shadow`/`prefer`/`only`)
  and the same result-verification gate from prompt 07 (a federated result is verified
  like any other — federation does not mean blind trust).
- Settlement: where a federated network has its own payment rail, the adapter handles
  paying it and the platform reconciles cost; where it can settle via x402, reuse prompt
  04. Never route a job to a network we can't actually pay.

### 2. The open SDK — `@three-ws/inference` (or fit the existing naming)

A published package (follow the exact build/publish process in `STRUCTURE.md` — versioned,
real, installable) that lets any developer or agent use the network:

- **Discover:** list models, nodes, prices, regions (wraps `GET /api/oin/nodes` + pricing).
- **Infer / generate:** one call to run a job (LLM completion, text→3D, retexture, etc.)
  that pays via x402 under the hood (reuse the x402 client patterns already in the repo),
  routes through the mesh, verifies the signed result, and returns it. Same ergonomics as
  calling a centralized API — that's the point.
- **Operate:** thin helpers for node operators (register, heartbeat, claim) so the node
  client (02) and third-party node software can share one client library.
- Typed, documented, with a real runnable example in `examples/` and docs in `docs/`.
  Every code sample uses real endpoints — no pseudo-code.

### 3. Wire it into the platform's developer story

- Add the SDK + network to the docs/developer surface (`docs/`, the SDKs page) and the
  bazaar catalog so other agents discover "decentralized compute" as a payable service.
- An MCP tool / endpoint so agents in the three.ws ecosystem can call the mesh
  conversationally (match the existing `mcp-server` / `api/x402/*` patterns).

## Hard requirements

- **Only real, callable networks.** Every federation adapter hits a real API and can
  actually run a job and be paid. An adapter you can't exercise end-to-end doesn't ship.
- **Federated results are verified** (prompt 07) and **priced/settled** (prompt 04) like
  native ones — same trust model, same money model.
- **$THREE / USDC only** in pricing, payouts, SDK, and docs. The Livepeer leg keeps
  working; no other token is referenced anywhere.
- SDK is **published for real** per `STRUCTURE.md`, not a local-only folder.

## Definition of done

- The Livepeer adapter and at least one more real network are routable through the
  existing router under policy, with verification + settlement applied.
- The side-by-side inference comparison demo still works (now via the federation adapter).
- `@three-ws/inference` (or the chosen name) is published, installable, and a real
  `examples/` script runs a job end-to-end (discover → pay → infer → verified result).
- An MCP tool exposes the network to ecosystem agents; it's in the bazaar catalog and docs.
- Tests cover the federation adapter contract, policy routing to a federated network, and
  the SDK's discover/infer path (mockable transport at the test boundary only — the
  shipped code calls real APIs). Existing tests pass.
- `data/changelog.json` entries (SDK release + federation), `npm run build:pages`,
  `npm test` green.
- Explicit-path staging; reviewed `git diff`; no bundled `api/*.js`.

Federation gives the mesh capacity; the SDK gives it reach. Ship both real.
