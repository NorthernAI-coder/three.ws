# Phase 4 · 01 — Node Registry, Protocol & Schema (the spine)

You are a senior infrastructure engineer on **three.ws**. Read `CLAUDE.md` and
`STRUCTURE.md` first — CLAUDE.md overrides defaults. This is the **foundation** of
the Open Inference Network: the protocol, the database schema, and the onchain
registry that prompts 02–09 all build on. Build it so completely that the node
client, the router, settlement, and the operator console can each plug in without
redefining anything.

## What you're building

The shared spine of the decentralized GPU mesh:

1. **The protocol (a versioned spec doc + shared JS module).** Define, precisely:
   - **Capability advertisement** — what a node tells the network it can serve:
     `{ node_id, operator_addr, models:[{ key, hardware, max_batch }], gpu:{ name, vram_gb }, regions, version, max_concurrency }`. Model keys reuse the existing worker
     discriminators (`hunyuan3d`, `trellis`, `triposr`, `triposg`, `texture`, `rembg`,
     `segment`, `remesh`, `text2motion`, `longcat`, `unirig`, and an `llm:<model>` form
     for the inference leg). Don't invent a parallel taxonomy — mirror `workers/README.md`
     and `api/_providers/gcp.js`.
   - **Job manifest** — what the network hands a node: `{ job_id, model_key, inputs,
     params, max_price_atomics, deadline_ms, nonce }`, matching the existing worker
     request bodies (`POST /infer`, `/reconstruct`, etc.).
   - **Signed response** — what a node returns: the result envelope plus a node
     signature over `keccak256(job_id || model_key || result_hash || node_id || nonce)`.
     Define the canonical serialization once so the node client (02), the router (03),
     and verification (07) all hash identically. Support both an EVM (secp256k1) and a
     Solana (ed25519) node key so operators on either chain can serve.
   - **Receipt** — the settlement artifact: `{ job_id, node_id, payer, model_key,
     price_atomics, result_hash, node_sig, ts }`, EIP-712 / JWS shaped to match
     `api/_lib/x402-spec.js` so prompt 04 can settle it directly.
   Write the spec as `docs/open-inference-protocol.md` AND a single importable module
   `api/_lib/oin/protocol.js` (canonical hashing, serialization, sig verify helpers)
   so code and docs can never drift.

2. **The database schema.** New timestamped migration in `api/_lib/migrations/` (and
   mirror into `api/_lib/schema.sql`):
   - `oin_nodes` — registered nodes: id, operator address (+ chain), payout address,
     advertised capabilities (jsonb), endpoint URL (or "pull" flag for poll-based
     nodes), status, version, registered/last-seen timestamps, onchain registration ref.
   - `oin_jobs` — every dispatched job: id, model_key, requester (agent/user), assigned
     node_id, status (`pending|claimed|running|done|failed|reassigned`), price_atomics,
     result_hash, node_sig, latency_ms, fallback_used (bool), created/settled timestamps.
   - `oin_node_heartbeats` — last-seen, load, healthy-models, latency samples (feeds the
     router in 03 and reputation in 07).
   Index for the router's hot path: "healthy nodes advertising model X, ranked." Every
   table has the columns reputation (07) and settlement (04) will need — design for them
   now so those prompts don't re-migrate.

3. **The onchain registry.** A node registers its identity onchain so the roster is
   provable, not a database we could fake. Reuse the **existing** identity rails — do
   NOT write a new bespoke chain integration from scratch:
   - EVM: extend the ERC-8004 / reputation-registry pattern from Phase 3 (`contracts/`)
     with a `NodeRegistry` entry (operator → node pubkey, payout addr, capability hash,
     stake placeholder for 07). Solana: a Metaplex Core asset or PDA mirroring it.
   - On-chain stores the *commitment* (capability hash, payout address, operator);
     the full capability JSON lives off-chain in `oin_nodes`, verified against the hash.
   If deploying a new contract is out of scope for one chat, ship the contract source +
   ABI + a register/read helper in `api/_lib/oin/registry.js` that reads/writes the real
   chain, and document the deploy step. Do not stub the chain call.

4. **Registration & roster APIs:**
   - `POST /api/oin/register` — a node (or the operator console in 06) registers:
     validates the capability advert, verifies the onchain registration ref, writes
     `oin_nodes`. Idempotent on node_id.
   - `POST /api/oin/heartbeat` — node liveness + load + healthy-model report.
   - `GET /api/oin/nodes` — public roster (the network's "who's online"): healthy nodes,
     models served, region, reputation placeholder, last seen. Powers 06's network map.
   - `GET /api/oin/nodes/:id` — single node detail.
   Auth: a node authenticates with its node key (signature challenge), not a shared
   secret — the whole point is no central trust. Use `api/_lib/http.js` helpers and
   `api/_lib/rate-limit.js`.

## Definition of done

- `docs/open-inference-protocol.md` is precise enough that prompt 02 implements a node
  against it with zero further questions; `api/_lib/oin/protocol.js` is its executable form.
- Migration applies cleanly; `schema.sql` matches; tables indexed for the router's query.
- A node can register against the **real chain** and appear in `GET /api/oin/nodes`.
- Signature verify round-trips for both EVM and Solana node keys (unit tests in
  `tests/` covering the canonical hash, both key types, and a tampered-result rejection).
- Registration/heartbeat/roster endpoints handle every state; bad adverts rejected with
  actionable errors; rate-limited.
- `data/changelog.json` entry (developer/infra-facing: "Open Inference Network — node
  registry & protocol"), `npm run build:pages`, `npm test` green.
- `git diff` reviewed; explicit-path staging; no bundled `api/*.js`.

Build the spine right. Everything else in Phase 4 leans on it.
