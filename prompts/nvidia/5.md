# Phase 4 · 05 — Route the Real Workloads onto the Mesh

You are a senior backend engineer on **three.ws**. Read `CLAUDE.md` and `STRUCTURE.md`
first — CLAUDE.md overrides defaults. This is where the network stops being
infrastructure and starts carrying **real production traffic**. Prompts 01–04 built the
spine, the node, the router, and settlement; now wire the platform's actual GPU jobs
through it — safely, behind policy, with the centralized path as a guaranteed fallback.

Read `api/forge.js`, `api/_lib/forge-tiers.js`, `api/_providers/gcp.js`, the avatar
pipeline (`workers/avatar-pipeline-controller`, the `reconstruct` mode in `gcp.js`),
`api/inference/livepeer.js`, and the router/policy from prompt 03.

## What you're building

The integration that routes existing workloads to the mesh per the routing policy, so
that — over time — the README north star is met: **≥50% of production GPU traffic served
by independent node operators.** You are not changing what these features do; you're
changing *where the GPU runs*, invisibly, with no regression.

### Wire these call sites (each behind its model's routing policy)

1. **Text/image → 3D mesh (`api/forge.js`).** The biggest GPU consumer. Where `forge`
   today dispatches to `gcp.js` (`hunyuan3d`/`trellis`/`triposr`/`triposg`), let it
   choose `decentralized.js` per `api/_lib/oin/policy.js`. `prefer` → mesh first, GCP
   fallback. The forge tier (draft/standard/high) maps to the node hardware/price tier
   from prompt 04. No change to the forge API contract or the client.

2. **Avatar reconstruction & rigging.** The `reconstruct` / `rerig` modes. These are
   multi-step (mesh backend → UniRig); route the GPU-heavy sub-steps to the mesh where a
   node advertises them, keep the CPU controller centralized. The README explicitly
   calls out offloading "the heaviest step (multi-view face fitting) onto external GPU
   nodes" — deliver that.

3. **Texture / stylize / rembg / segment / remesh / text2motion.** Same pattern via
   their `gcp.js` modes. Start each at `shadow`, promote to `prefer` as nodes prove out.

4. **LLM inference (`api/inference/livepeer.js`).** Generalize the existing Livepeer
   experiment: the inference router can send a completion to a mesh node advertising
   `llm:<model>` (priced per-token via prompt 04's `upto` scheme), with the centralized
   LLM path as fallback. Keep the side-by-side comparison demo working.

### Rollout safety (this is the whole job — do it right)

- **Default to `shadow` for every newly-wired model.** Shadow runs the job on the mesh
  in parallel, serves the centralized result, and records mesh-vs-centralized latency,
  success, and result-quality deltas. Only after the data looks good does an operator
  flip the model to `prefer`. Never flip a model straight to `only`.
- **The user never regresses.** Every wired path keeps its centralized fallback. A mesh
  miss/timeout/bad-sig is invisible to the user (the router handles it; you just pass the
  policy through).
- **Per-model kill switch.** Any model can be reverted to `off` (centralized only) via
  policy without a deploy.
- **Traffic accounting.** Record, per job, whether it was mesh-served or centralized, so
  the network's "% served by independent nodes" is measurable — this feeds the operator
  console (06) and QA (09).

## Hard requirements

- **No behavior change for the user** beyond (ideally) latency/cost — same outputs, same
  API contracts, same client code.
- **No mock routing** — real policy table, real provider selection, real fallback.
- Shadow mode must not bill the user twice or settle two payouts for one user-facing
  result — settle only the served result; a shadow run that isn't served either isn't
  settled or is settled from a separate evaluation budget (be explicit which).

## Definition of done

- `forge`, avatar reconstruction, the edit workers, and LLM inference each route through
  `decentralized.js` under policy, with GCP/centralized fallback intact.
- A real forge generation completes on a registered node (policy `prefer`) with identical
  output quality to the centralized path; with nodes off, the same generation succeeds
  centrally.
- Shadow mode records comparison metrics without affecting the user or double-billing.
- Per-model policy flip (`shadow`→`prefer`→`off`) takes effect with no deploy.
- Tests: each wired call site selects provider per policy and falls back correctly;
  traffic-split accounting is recorded.
- `data/changelog.json` entry ("3D generation can now run on the decentralized GPU
  mesh"), `npm run build:pages`, `npm test` green.
- Explicit-path staging; reviewed `git diff`; no bundled `api/*.js`.

When a forge job silently runs on someone else's H100 in another country and the user
just sees their avatar appear, Phase 4 is real.
