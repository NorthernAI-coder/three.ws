# 06 — Premium vanity-address inventory on spot CPU

## Mission

The platform's `vanity_grinder` MCP tool grinds Solana vanity addresses **in-process**, capped
at 1.5M iterations, $0.05/call — so 4–5+ character prefixes are unreachable. Build a batch
grinding pipeline on cheap GCP spot CPU (credits), an encrypted inventory store, and a
"premium vanity" product tier that sells from inventory instantly. Durable, sellable asset;
near-zero credit cost. This handles **private keys** — security is the hard part of this
prompt, not the grinding.

## Prerequisites

- Prompt 01 ran (project, `gcloud`, Compute API enabled).

## Context (from prior code audit; re-verify)

- MCP tool: `mcp-server/src/tools/vanity-grinder.js`; grind routines
  `mcp-server/src/lib/pump-vanity.js` (raw keypair) and `mcp-server/src/lib/mnemonic-vanity.js`
  (BIP-39). `MAX_ITERATIONS_CAP` ≈ 1.5M with an up-front difficulty guard. Price env
  `MCP_VANITY_PRICE_USD` ($0.05, x402 `exact` on Solana).
- **A Rust implementation exists:** `crates/vanity-grinder/` (+ WASM build under
  `src/solana/vanity/`). Rust grinds orders of magnitude faster than the JS loop — use it for
  the batch pipeline.
- x402 payment plumbing and USDC settlement already exist platform-wide (see other
  `api/x402/*` endpoints for the pattern).
- Study how the platform handles secrets/wallet keys today (search `api/_lib/` for encryption
  patterns, KMS usage, env-key encryption) and match the strongest existing pattern; if none is
  strong enough for stored private keys, use GCP KMS envelope encryption and justify it.

## Tasks

1. **Batch grinder job.** Package `crates/vanity-grinder` as a container (Cloud Run **Jobs**,
   CPU-only, or a GCE spot MIG — pick per cost/simplicity and justify). Input: a target list of
   prefixes/suffixes (case-sensitive and insensitive variants). Output: found keypairs written
   **directly in encrypted form** (never plaintext at rest, never in logs — encrypt in-process
   before any write). Compute a target list worth grinding: 3–4 char brandable prefixes
   (`SOL…`, `PUMP…`, `AGNT…`, `MOON…`, common names), plus `pump`-suffix addresses if the
   pump.fun mint flow can consume supplied keypairs (check the existing launcher plumbing —
   generic mint-at-runtime plumbing is fine; do not hardcode third-party coin references).
2. **Inventory store.** A table (follow the platform's existing DB conventions/migrations in
   `api/_lib/migrations/`) holding: address (public), pattern, difficulty score, KMS-encrypted
   secret payload, status (`available`/`reserved`/`sold`/`revealed`), price, timestamps,
   purchaser identity. Decryption only at the moment of delivery.
3. **Product surface:**
   - Extend the `vanity_grinder` MCP tool (and/or a new `api/x402/vanity-premium.js` endpoint)
     with an inventory tier: list available patterns + prices, purchase via x402 (price scaled
     by difficulty — derive a curve from grind cost, e.g. $1–$50), deliver the key **once**,
     mark `revealed`, and never serve it again. Keep the existing $0.05 live-grind tier
     untouched.
   - A browsable listing page/endpoint consistent with existing marketplace patterns so the
     inventory is discoverable (wire navigation — no orphan pages).
4. **Custody honesty (required product copy):** the platform generated these keys, so a buyer
   must be told plainly: "this key was generated by three.ws; for high-value use, transfer
   assets to a self-generated wallet or use it as a token mint address, not a treasury."
   Put this in the delivery payload and the listing UI. Also implement post-delivery secret
   destruction as an option (delete ciphertext after reveal + configurable retention),
   defaulting to delete-after-reveal.
5. **Run a real batch.** Grind an initial inventory (e.g. 100–300 addresses across the target
   list) on spot instances. Record throughput (keys/sec/vCPU), $/address at each difficulty,
   and load the inventory.
6. **E2E verification:** buy one from inventory through the real x402 flow on a test wallet;
   confirm single delivery, `revealed` state, ciphertext destruction, and that the address
   works (sign a transaction with the delivered key on devnet or verify keypair→address
   locally).
7. **Security self-review** (write it up): threat model — DB dump (keys are KMS-encrypted),
   log leakage (grep the pipeline for any secret logging), replay of delivery endpoint
   (single-use enforced server-side), insider access (KMS IAM restricted to the delivery
   service identity). Fix anything found.

## Guardrails

- Plaintext secrets: never at rest, never in logs, never in git, never in analytics.
- The commit gate: address patterns and copy must not reference third-party crypto projects.
  Brandable generic prefixes only; `$THREE`-related patterns welcome.
- Spot instances get preempted — the grinder must checkpoint its target list and be re-runnable.
